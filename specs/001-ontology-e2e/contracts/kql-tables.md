# KQL Table Contracts

**Date**: 2026-04-01 | **Updated**: 2026-04-03 | **Plan**: [../plan.md](../plan.md)

**IMPORTANT**: The schemas below reflect the **actual deployed tables** as of 2026-04-03. They differ from the original design — column names were aligned with ontology TimeSeries data bindings.

## SensorTelemetry

Real-time sensor readings from IoT sensors attached to manufacturing equipment.

```kql
.create-merge table SensorTelemetry (
    SensorId: string,
    Timestamp: datetime,
    Value: real,
    Quality: string
)
```

| Column | Type | Description | Example |
|--------|------|-------------|---------|
| SensorId | string | FK to DIM_SENSOR.SensorId / Ontology Sensor.SensorId | `SG-SN-001` |
| Timestamp | datetime | UTC measurement time | `2026-04-01T10:30:00Z` |
| Value | real | Measured value (units vary per sensor) | `1085.3` |
| Quality | string | Signal quality | `Good`, `Fair`, `Poor` |

**Note**: `Unit` column removed — unit info comes from ontology Sensor entity (Sensor.Unit property).

**Ontology TimeSeries Binding**: Sensor entity → `Value`, `Quality` properties

**Ingestion Rate**: ~20 rows per 30-minute interval (one per sensor)

## EquipmentStatus

Periodic equipment operational state snapshots.

```kql
.create-merge table EquipmentStatus (
    EquipmentId: string,
    Timestamp: datetime,
    RunTimeMinutes: real,
    DownTimeMinutes: real,
    OperatingStatus: string
)
```

| Column | Type | Description | Example |
|--------|------|-------------|---------|
| EquipmentId | string | FK to DIM_EQUIPMENT.EquipmentId / Ontology Equipment.EquipmentId | `SG-EQ-001` |
| Timestamp | datetime | UTC snapshot time | `2026-04-01T10:30:00Z` |
| RunTimeMinutes | real | Cumulative run minutes | `12450.5` |
| DownTimeMinutes | real | Cumulative downtime minutes | `150.7` |
| OperatingStatus | string | Current status | `Operating`, `Idle`, `Maintenance` |

**Note**: Columns renamed from original (`Status`→`OperatingStatus`, `RunHours`→`RunTimeMinutes`, `CycleCount`→`DownTimeMinutes`) to match ontology TimeSeries properties.

**Ontology TimeSeries Binding**: Equipment entity → `RunTimeMinutes`, `DownTimeMinutes`, `OperatingStatus` properties

**Ingestion Rate**: ~18 rows per 30-minute interval (one per equipment)

---

## ProductionMetrics

Production line output and efficiency metrics.

```kql
.create-merge table ProductionMetrics (
    LineId: string,
    Timestamp: datetime,
    Efficiency: real,
    UnitCount: long,
    CycleTime: real
)
```

| Column | Type | Description | Example |
|--------|------|-------------|---------|
| LineId | string | FK to DIM_LINE.LineId / Ontology ProductionLine.LineId | `SG-LN-001` |
| Timestamp | datetime | UTC measurement time | `2026-04-01T10:30:00Z` |
| Efficiency | real | Line efficiency (0.0-1.0) | `0.847` |
| UnitCount | long | Units produced in interval | `45` |
| CycleTime | real | Average cycle time in seconds | `32.5` |

**Note**: Columns renamed from original (`OutputUnits`→`UnitCount`, `DefectRate`→`Efficiency`, `OEE`→`CycleTime`) to match ontology TimeSeries properties.

**Ontology TimeSeries Binding**: ProductionLine entity → `Efficiency`, `UnitCount`, `CycleTime` properties

**Ingestion Rate**: ~12 rows per 30-minute interval (one per line)

## Alerts

Operational alerts generated by threshold violations or anomaly detection.

```kql
.create-merge table Alerts (
    AlertId: string,
    EntityId: string,
    EntityType: string,
    Timestamp: datetime,
    Severity: string,
    Message: string,
    IsResolved: bool
)
```

| Column | Type | Description | Example |
|--------|------|-------------|---------|
| AlertId | string | Unique alert ID | `ALT-20260401-001` |
| EntityId | string | ID of entity that triggered alert | `SG-SN-001` |
| EntityType | string | Type of entity | `Sensor`, `Equipment`, `Line` |
| Timestamp | datetime | UTC alert time | `2026-04-01T10:30:00Z` |
| Severity | string | Alert severity | `Critical`, `Warning`, `Info` |
| Message | string | Alert description | `Temperature exceeded threshold` |
| IsResolved | bool | Resolution status | `false` |

**Ingestion Rate**: ~5% probability per interval, 2-3 alerts per interval on average
